The figure below illustrates the overall runtime architecture of CARP Mobile Sensing (CAMS). The architecture consists of three main layers:

| Layer | Main responsibility |
| --- | --- |
| Client manager layer | The main runtime layer of CAMS - orchestrates study lifecycle, runtime controllers, and registries. |
| Service layer | Provides deployment configuration and data persistence/upload services. |
| Sampling layer | Collects measurements through sampling packages and probes. |

CAMS builds on top of a set of Flutter plugins that access sensors and services in the underlying OS, external wearable devices, and cloud-based services.
This architecture is designed to achieve the non-functional software architecture goals of being highly extensible, cross-platform, and maintainable.

## Client manager layer

The [SmartPhoneClientManager](https://pub.dev/documentation/carp_mobile_sensing/latest/runtime/SmartPhoneClientManager-class.html) is the core runtime of CAMS. It is a singleton and holds registries, controllers, and services that support data sampling:

  | Component | Role |
  | --- | --- |
  | [DeploymentService](https://pub.dev/documentation/carp_core/latest/carp_core_deployment/DeploymentService-class.html) | Handles study deployment configurations for the client. |
  | [DataManagerRegistry](https://pub.dev/documentation/carp_mobile_sensing/latest/runtime/DataManagerRegistry-class.html) | Registry of available [DataManager](https://pub.dev/documentation/carp_mobile_sensing/latest/runtime/DataManager-class.html) implementations. |
  | [SamplingPackageRegistry](https://pub.dev/documentation/carp_mobile_sensing/latest/runtime/SamplingPackageRegistry-class.html) | Registry of available [SamplingPackage](https://pub.dev/documentation/carp_mobile_sensing/latest/runtime/SamplingPackage-class.html) implementations. |
  | [DataTransformerSchemaRegistry](https://pub.dev/documentation/carp_mobile_sensing/latest/runtime/DataTransformerSchemaRegistry-class.html) | Registry of available [DataTransformerSchema](https://pub.dev/documentation/carp_mobile_sensing/latest/domain/DataTransformerSchema-class.html) mappings. |
  | [DeviceController](https://pub.dev/documentation/carp_mobile_sensing/latest/runtime/DeviceController-class.html) | Controls connected devices during runtime. |
  | [AppTaskController](https://pub.dev/documentation/carp_mobile_sensing/latest/runtime/AppTaskController-class.html) | Manages [AppTask](https://pub.dev/documentation/carp_mobile_sensing/latest/domain/AppTask-class.html) lifecycle and queueing. |
  | [NotificationController](https://pub.dev/documentation/carp_mobile_sensing/latest/runtime/NotificationController-class.html) | Handles OS-level notifications. |

* The SmartPhoneClientManager can be [configured](https://pub.dev/documentation/carp_mobile_sensing/latest/runtime/SmartPhoneClientManager/configure.html) by specifying deployment service, device controller, notification controller, and related runtime options.

* Study deployments that should run on a SmartPhoneClientManager can be [added](https://pub.dev/documentation/carp_mobile_sensing/latest/runtime/SmartPhoneClientManager/addStudy.html), which creates and hosts a [SmartphoneDeploymentController](https://pub.dev/documentation/carp_mobile_sensing/latest/runtime/SmartphoneDeploymentController-class.html) per study.

  * The SmartphoneDeploymentController can be [configured](https://pub.dev/documentation/carp_mobile_sensing/latest/runtime/SmartphoneDeploymentController/configure.html) by optionally specifying a [DataEndPoint](https://pub.dev/documentation/carp_core/latest/carp_core_deployment/DataEndPoint-class.html) and a [DataTransformer](https://pub.dev/documentation/carp_mobile_sensing/latest/domain/DataTransformer.html).

  * The SmartphoneDeploymentController uses the set of registries in the SmartPhoneClientManager to look up an appropriate data manager, relevant sampling packages, data transformers, and sensing probes for executing the sensing study deployment protocol.

When the SmartPhoneClientManager has been configured and studies have been added, data sampling can be controlled by [starting](https://pub.dev/documentation/carp_mobile_sensing/latest/runtime/SmartPhoneClientManager/start.html) or [stopping](https://pub.dev/documentation/carp_mobile_sensing/latest/runtime/SmartPhoneClientManager/stop.html) all studies on a client. This can also be done per study, by using the [start](https://pub.dev/documentation/carp_mobile_sensing/latest/runtime/SmartphoneDeploymentController/start.html) and [stop](https://pub.dev/documentation/carp_mobile_sensing/latest/runtime/SmartphoneDeploymentController/stop.html) methods on `SmartphoneDeploymentController`.

## Service layer

The service layer (top layer) holds

* The [Deployment Service](https://pub.dev/documentation/carp_core/latest/carp_core_deployment/DeploymentService-class.html), which is able to manage and deploy the [StudyProtocol](https://pub.dev/documentation/carp_core/latest/carp_core_protocols/StudyProtocol-class.html).

* [Data Managers](/carp-mobile-sensing/data-managers), which know how to store, save, or upload data, such as the [SQLiteDataManager](https://pub.dev/documentation/carp_mobile_sensing/latest/data_managers/SQLiteDataManager-class.html).

## Sampling layer

The sampling layer (bottom layer) holds

* A set of [SamplingPackage](https://pub.dev/documentation/carp_mobile_sensing/latest/runtime/SamplingPackage-class.html)s, which can collect a specific [type of measure](/carp-mobile-sensing/measure-types). CAMS comes with two built-in sampling packages ([DeviceSamplingPackage](https://pub.dev/documentation/carp_mobile_sensing/latest/device/DeviceSamplingPackage-class.html) and [SensorSamplingPackage](https://pub.dev/documentation/carp_mobile_sensing/latest/sensors/SensorSamplingPackage-class.html)), but most sampling packages are external and included as needed. In this way, an app only requests permissions for the probes and sensors it actually uses. A sampling package can also collect data from external devices (for example [Polar heart rate monitors](https://pub.dev/packages/carp_polar_package)); in this case it includes a [DeviceManager](https://pub.dev/documentation/carp_mobile_sensing/latest/runtime/DeviceManager-class.html). Such devices can also be online services, such as [WeatherServiceManager](https://pub.dev/documentation/carp_context_package/latest/carp_context_package/WeatherServiceManager-class.html). The [carp_context_package](https://pub.dev/packages/carp_context_package) is an example of an externally loaded sampling package available on [pub.dev](https://pub.dev/).

* Each sampling package contains one or more [Probe](https://pub.dev/documentation/carp_mobile_sensing/latest/runtime/Probe-class.html)s that access underlying sensors and collect data. Access to sensor data is typically done via Flutter plugins. These plugins are external to CAMS and loaded by the sampling package as needed. For example, [carp_context_package](https://pub.dev/packages/carp_context_package) uses the [weather](https://pub.dev/packages/weather) plugin to access weather data.
